## 1. Бизнес-контекст

* **Ситуация:** облачный провайдер предоставляет Virtual Machines (ВМ), Managed Kubernetes и Managed PostgreSQL.
* **Проблема:** отсутствие прозрачности для клиентов
* **Цель:** предоставить клиентам инструменты самообслуживания в Личном кабинете (графики), а инженерам — единую консоль для траблшутинга

---

## 2. Функциональные требования

### 2.1. Сбор данных

| ID | Требование | Описание и Критерии приёмки |
| :--- | :--- | :--- |
| **FR-COL-01** | Сбор базовых метрик | Система должна собирать метрики утилизации ресурсов: `CPU`, `RAM`, `Disk`, `Network`. |
| **FR-COL-02** | Охват продуктов | Сбор метрик должен осуществляться с трех типов ресурсов: ВМ клиентов, ноды Managed K8s, инстансы Managed PostgreSQL. |
| **FR-COL-03** | Сбор логов (Фильтрация) | Система должна собирать логи **исключительно** `ERROR` и `WARN`. Логи уровня `INFO` и `DEBUG` должны отбрасываться на стороне агента для экономии бюджета. |
| **FR-COL-04** | Тегирование (Multi-tenancy) | Каждая метрика и строка лога, отправляемая в хранилище, должна быть жестко размечена тегами: `tenant_id` (ID клиента) и `resource_id` (ID ресурса). |

### 2.2. Хранение данных
*Реализуется средствами ClickHouse TTL.*

| ID | Требование | Описание и Критерии приёмки |
| :--- | :--- | :--- |
| **FR-STO-01** | Хранение метрик | Все собранные метрики должны храниться в базе данных ровно **1 месяц**, после чего автоматически удаляться (срок хранения требует уточнения). |
| **FR-STO-02** | Хранение логов | Текстовые логи (`ERROR`, `WARN`) должны храниться ровно **1 неделю**, после чего автоматически удаляться (срок хранения требует уточнения). |
| **FR-STO-03** | Хранение метаданных | Настройки дашбордов, профили пользователей и маппинг `tenant_id` к ресурсам должны храниться в MongoDB. |

### 2.3. Доступ и Визуализация

| ID | Требование | Описание и Критерии приёмки |
| :--- | :--- | :--- |
| **FR-VIS-01** | Изоляция данных клиента | При запросе графиков из Личного кабинета, бэкенд системы обязан применять жесткую фильтрацию по `tenant_id`. Клиент не должен иметь технической возможности получить данные других арендаторов. |
| **FR-VIS-02** | Скорость обновления | Задержка между возникновением события на ресурсе и его появлением на графике в Личном кабинете не должна превышать **1-2 минуты**. |
| **FR-VIS-03** | Консоль инженеров | Инженеры поддержки должны иметь доступ к единому внутреннему UI (HyperDX), позволяющему выполнять сквозной поиск логов и метрик по всем ресурсам провайдера. |

### 2.4. Алертинг

| ID | Требование | Описание и Критерии приёмки |
| :--- | :--- | :--- |
| **FR-ALT-01** | Базовые уведомления | *(Отложено для v2.0)* Система должна уметь генерировать события при достижении порогов (например, Disk > 95%, CPU = 100% длительное время). В первой версии реализуется только сбор и отображение. |

## 3. Нефункциональные требования (Non-Functional Requirements)

* **NFR-SCA (Масштаб):** Система (ClickHouse + OTel Collector) должна выдерживать входящий поток телеметрии от 50 000 ВМ и 5 000 кластеров БД, принадлежащих 10 000 уникальным клиентам.
* **NFR-CST (Бюджет):** Затраты на хранение минимизируются за счет:
  1. Отказа от сбора INFO-логов.
  2. Использования столбцового сжатия ClickHouse.
  3. Короткого периода хранения логов (7 дней).

## 4. Что можно уточнить?

1. Дефолтный вид графиков для клиентов (последние 3 часа или последние 24 часа)?

2. Допустимо ли сглаживание старых данных (например, за прошлую неделю достаточно информации за 1 час)?

3. Нужны ли аналитические отчеты за месяц или достаточно отображения на графиках?

5. Длительность хранения логов и метрик? Если время разное, то нужно ли в интерфейсе явно ограничивать выбор периода для логов 7-ю днями, чтобы у клиента не возникало чувства ошибки (пустой экран)?

6. Нужна ли архивация данных? В течение какого срока (например, 2 дня) данные должны считаться 'горячими' и открываться мгновенно? Допустима ли задержка при доступе к 'холодным' данным (старше 2 дней)?

* при необходимости архивации - добавление S3 (Clickhouse + S3)
